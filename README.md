# 智能仓储调物流仿真系统(Smart Warehouse Logistics Simulation System)

本系统是一款基于 **C++17** 标准开发的高性能、多 AGV（Automated Guided Vehicle）智能仓储调度与路径规划仿真系统。系统完美融合了经典组合优化数学算法与现代时空路径搜索技术，在 **EasyX 图形库** 的驱动下，实现了高并发动态订单环境下的全局最优任务指派、多车协同无冲突时空寻路、以及丝滑的 20x20 仓储物流视觉演练大盘。

---

## 1. 📌项目背景与问题描述 (Background & Problem Description)

本系统模拟的是一个经典的**多智能体任务分配与路径规划（Multi-Agent Task Assignment and Path Finding, MATAPF）**问题。在一个抽象的 20x20 离散网格化仓储环境中，部署了多个静态货架、一个出货口（OUT站）以及 $N$ 辆可移动的 AGV 小车。

当系统处于海量动态订单流的高并发环境下，需要通过集中式控制中心解决以下三个层面的数学与工程问题：

### 1.1 任务指派中的局部最优与组合优化问题
* **问题描述**：当系统同时存在多个等待处理的订单和多辆闲置 AGV 时，若采用传统的“就近分配（贪心算法）”，在局部可能达到最优，但从全局来看，极易导致远端小车长期空闲、近端小车高频抢单，进而引发全局总完单耗时（Make-span）大幅增加。
* **数学建模**：该问题本质上是一个**二分图最优匹配（Assignment Problem）**。系统需要构建一个“小车-任务”的代价矩阵（Cost Matrix），在多对多的约束下，求解使全局总移动代价（或总等待时间）最小化的最优指派方案。

### 1.2 多智能体寻路中的时空冲突与死锁问题 (MAPF)
* **问题描述**：多辆 AGV 在同一地图内并发执行路径规划。如果仅依赖传统的静态二维寻路算法（如标准 A* 算法），小车只会关注空间障碍物（货架），而忽视了其他小车的动态位置。当多车在狭窄的货架通道内相遇时，会引发两种不可逆的时空冲突：
  1. **点冲突 (Vertex Collision)**：两辆小车在同一时间戳 $t$ 试图进入同一个网格 $(x, y)$。
  2. **边对撞 (Edge Collision)**：在时间戳 $t \to t+1$ 的过程中，A车从格子1移动到格子2，而B车同时从格子2移动到格子1，发生对头相撞。
* **技术挑战**：如何在寻路算法中引入**时间维度 $T$**，将搜索空间升维至 $(X, Y, Time)$ 三维时空，并在生成路径时强制规避这两种冲突，确保多车在拓扑网络中绝不。
发生停摆现象
### 1.3 离散逻辑推演与连续视觉渲染的映射误差
* **问题描述**：在底层算法逻辑中，时钟推进和路径搜索都是基于离散的“网格计数（Ticks）”进行的。如果直接将这种离散的状态反映在 GUI 界面上，小车将会产生跳跃、瞬移的卡顿感；同时，高频的全局刷新会导致屏幕剧烈闪烁。
* **工程挑战**：如何在不破坏底层离散算法纯净性的前提下，在表现层建立一套平滑的直线插值机制与双缓冲渲染机制，实现底层“离散 Tick 逻辑”向表层“连续匀速视觉”的丝滑映射。

## 2. 系统核心功能及特性 (Core Features & System Characteristics)

本系统围绕“调度-寻路-渲染”三大核心链路，构建了具备高健壮性、高表现力的完整仿真生态，核心特性表现为以下五个维度：

### 2.1 基于匈牙利算法的全局集中式调度
* **全局最优指派**：升级优化了原先传统局部贪心的就近派单机制。系统内置 `HungarianOptimizer` 数学优化器，在 WMS 触发派单时，实时采集所有空闲 AGV 与待处理订单的“时空代价”，构建动态代价矩阵（Cost Matrix）。通过增广路径归约，在 $O(N^3)$ 时间复杂度内求解出全局移动总代价最小的二分图完美匹配方案。
* **非方阵刚性补齐**：针对工业仿真中经常出现的“车少单多”或“车多单少”的非对称匹配场景，算法底层实现了动态方阵补齐机制，自动以极大值或虚拟节点填补空缺，确保调度逻辑在任意极端吞吐状态下均能稳定收敛。

### 2.2 时空三维路径规划 (Time-Windowed A* Algorithm)
* **时空动态避障**：寻路大脑 `Router` 将传统的二维空间寻路拓展为 $(X, Y, Time)$ 三维时空搜索，不仅将静态货架视为障碍，更将其他 AGV 未来的运动轨迹视为动态时空障碍。
* **全并发冲突防御**：依托全局时空预定表（`ReservationTable`），小车在选路时会强制进行安全性校验（`isTimeSpaceSafe`），从根源上刚性杜绝多车在狭窄通道内的**点冲突**与**边对撞（对头相撞）**。当预测到冲突时，小车会自动选择原地等待（Wait 状态）或绕行。
* **航向转弯惩罚机制**：搜索算法引入了方向变化惩罚权重（Turn Penalty），当路径发生非线性的方向跳变时施加代价惩罚，诱导小车规划出高直线度、少转弯的精简路径，大幅降低车辆在实际运行中的转向机械损耗。

### 2.3 离散型地图与连续的丝滑AGV小车运动渲染技术
* **离散逻辑与连续表象解耦**：底层算法完全基于离散的 Tick 时钟推演，而表现层 `Robot` 实体则引入了 `moveProgress` 浮点型分量直线插值机制。在离散的网格状态切换间，小车以平滑的像素步长进行匀速视觉过渡，消除AGV小车的移动卡顿感。
* **原生双缓冲无闪烁技术**：基于 EasyX 图形库并深度绑定 Win32 刷新机制，在 GUI 渲染主循环中开启 `BeginBatchDraw()` 批处理绘制。所有的背景、货架、路径虚线以及 AGV 状态图标均在内存缓冲区完成构图后一键物理提交，实现高频刷新下GUI界面的“零闪烁”的极佳视觉评审体验。

### 2.4 仓储环境与 ABC 货架热度可视化
* **ABC 货架立体热度渲染**：地图组件 `Map` 并非机械地涂抹色块，而是根据货架距离出货口（OUT站）的物理拓扑及访问频次，将整个仓储网络动态标定为 A类（核心频发区）、B类（温和过渡区）、C类（静谧低频区）。
* **3D 浮雕视觉传达**：利用色彩分量（RGB）的深浅错位，为货架实体精心渲染了 3D 浮雕阴影底框。高频热点区域呈现醒目的浮雕色彩，使评审老师能够直观、立体地洞察仓储布局的合理性。

### 2.5 交互式动态派单与高并发压力测试
* **人机交互印单控制**：系统支持双重接单模式。通过右侧看板的自动接单开关，用户可一键切换 WMS 状态。在手动模式下，支持鼠标点击选派的AGV小车和货架执行“定向人工印单”；在自动模式下，系统通过 Mersenne Twister 引擎自动生成防重复的随机订单流。
* **一键极端压测与效能大盘**：系统支持通过快捷键P一键触发极端压力测试（`isStressTesting`）。按下后即可瞬间向系统注入海量高并发订单，逼迫所有 AGV 进入满载极限工作状态，同时全局效能大盘实时计算并刷新**系统总吞吐量**与**平均订单生存周期**，用量化数据自证系统的调度极限。
* **简单测试快捷键**： 按下T键后可以触发自动生成4个随机订单，可用于简单测试系统

### 3.2 系统时钟推演与数据流向 (Dynamic Control Flow)
* **⚡ Step 1: WMS 主时钟 Tick 推进**
    * 系统时间戳自增，触发全盘生命周期更新。
* **📦 Step 2: OrderSystem 订单流注入**
    * *自动接单状态*：系统自动通过梅森旋转算法高频生成动态防重复订单。
    * *手动干预状态*：若发生鼠标点击货架事件，则触发定向人工即时印单。
* **🎯 Step 3: WarehouseManager 全局指派**
    * 核心调度系统实时收集当前所有待命小车与活跃订单，提取它们之间的拓扑代价。
    * 数据包送入 `HungarianOptimizer` 进行矩阵方阵刚性补齐，通过 $O(N^3)$ 匈牙利算法求解出全局总代价最小的完美指派对。
* **🧭 Step 4: Router 时空路径规划**
    * 获取任务的小车被唤醒，寻路大脑为其调用三维时空 A* 算法。
    * 算法高频检索 `ReservationTable`（全局时空预定表），通过锁定 `(X, Y, Time)` 的动态格子占有权，刚性阻断多车之间的点冲突与边对撞。
* **🤖 Step 5: Robot 实体状态机执行**
    * 小车成功获取无冲突时空路径序列，状态机切入工作状态。
    * 小车内部引入 `moveProgress` 浮点分量直线匀速插值，在离散网格状态切换之间实现像素级的平滑位移。
* **🎨 Step 6: GUI 双缓冲画布物理渲染**
    * 在每一帧渲染循环中，首先触发 `BeginBatchDraw()` 开启显存批处理锁。
    * 在内存缓冲区一口气完成静态地图、3D 浮雕热度货架、小车实时位置、路径轨迹虚线以及右侧看板大盘的重绘构图。
    * 最后执行 `EndBatchDraw()` 一键物理提交到屏幕，彻底终结重绘闪烁。

## 4. 核心算法解析 (Core Algorithm Highlights)

本系统的核心竞争力在于底层算法的自主实现与深度改良。系统拒绝调用任何第三方黑盒算法库，纯粹基于 C++ 标准容器完成了工业级调度与寻路算法的鲁棒落地。

### 4.1 三维时空 A* 寻路算法 (Time-Windowed A*)

传统的二维 A* 算法仅在空间维度 $(X, Y)$ 展开搜索，无法感知动态障碍物在时间轴上的变化。本系统将寻路状态升维至三维时空节点 $S = (X, Y, T)$，通过独创的**全局时空预定表 (Reservation Table)** 实现了多车并发无冲突路径规划。

#### 1. 动态冲突的数学判定准则
寻路在大脑 `Router` 在进行邻域拓扑拓展时，会高频调用 `isTimeSpaceSafe` 接口，针对以下两种多智能体致命冲突进行刚性拦截：
* **点冲突 (Vertex Collision)**：
  若当前寻路小车试图在时间戳 $t$ 进入坐标 $(x, y)$，但时空表中该格子已被其余小车预定，即：
  $$\text{Table}[x][y][t] \neq \text{IDLE}$$
  则判定为点冲突，立即熔断该分支。
* **边对撞冲突 (Edge Collision)**：
  当小车试图在 $t \to t+1$ 时刻从小格子 $A(x_1, y_1)$ 移动到相邻格子 $B(x_2, y_2)$，而时空表检测到另一辆小车正试图在同一时间段内从 $B$ 逆向行驶到 $A$，即：
  $$\text{Table}[A \to B][t] \cap \text{Table}[B \to A][t] \neq \emptyset$$
  判定为边对撞（对头相撞），算法强行拦截并迫使小车执行原地等待（Wait 状态）或绕行。

#### 2. 航向转弯惩罚机制 (Turn Penalty Optimization)
离散网格寻路极易产生无意义的“蛇形 Z 字走位”，这在实际工业中会造成 AGV 电机频繁正反转，加速机械磨损。
本系统在 `Router.cpp` 的代价推演中引入了航向惩罚项：

```cpp
// 当方向确实改变，且上一步不是原地等待，施加惩罚让路径尽量笔直不跳变
if ((lastDx != currentDx || lastDy != currentDy) && (lastDx != 0 || lastDy != 0)) {
    turnPenalty = 3;
}
int newG = current.g + edgeCost + turnPenalty;
```

通过为转向节点强行追加权重惩罚 $\Delta g = 3$，诱导启发式函数优先选择直线拓扑，搜索出的路径更符合真实工业车辆的动力学特性。

### 4.2 集中式匈牙利动态调度算法 (Hungarian Assignment Optimizer)

当 WMS 电商前台高频吐出大批订单时，系统不采用“先到先得”的脆弱机制，而是通过 `HungarianOptimizer` 求解二分图完美匹配，实现全局总完单代价的最优化。

#### 1. 算法核心步骤
优化器通过标准的矩阵归约逻辑，在 $O(N^3)$ 的时间复杂度内完成收敛：
1. **行归约 (Row Reduction)**：遍历代价矩阵，使每一行元素减去该行的绝对最小值，确保每行至少产生一个零元。
2. **列归约 (Column Reduction)**：遍历矩阵的每一列，减去该列的绝对最小值。
3. **最少覆盖线划零**：利用最少数量的水平或垂直线覆盖矩阵中所有的零。若覆盖线数等于矩阵阶数，则成功抓取最优解；若小于阶数，则动态调整未覆盖元素的阈值并循环迭代。

#### 2. 非方阵动态刚性补齐 (Non-Square Matrix Padding)
标准的匈牙利算法要求输入代价矩阵必须为 $N \times N$ 完美方阵。但在仿真运行中，“车少单多”（行数 $<$ 列数）或“车多单少”（行数 $>$ 列数）才是常态。
系统底层实现了 `convertToSquare` 鲁棒对齐机制：

```cpp
int maxDim = std::max(nRows, nCols);
// 自动以最大维度为基准，将非对称的“车-单”矩阵补齐为完美方阵
// 空缺的虚拟槽位代价刚性填0或安全极大值，确保增广路径树不会因维度失衡而发生指针悬空或逻辑死锁
convertToSquare(costMatrix, matrix, maxDim);
```

在最终映射指派池时，算法会自动通过 `if (matchRow[i] < nCols)` 过滤掉为了补齐方阵而生成的虚拟订单，完美实现了“动态吞吐、安全过滤”的工程闭环。

### 5.2 依赖库配置 (EasyX 安装指南)

本系统表现层使用 **EasyX Graphics Library** 进行窗口托管理与大盘渲染。配置步骤如下：

1. **下载安装包**：前往 [EasyX 官网 (easyx.cn)](https://easyx.cn) 下载最新版本的安装程序。
2. **一键集成**：运行下载的 `Setup.exe`，安装程序会自动检测您系统中已安装的 Visual Studio 版本。
3. **点击安装**：选择您当前开发使用的 Visual Studio（如 VS2022），点击对应的“安装”按钮。程序会自动将 `easyx.h` 和 `graphics.h` 释放至对应的系统包含路径（Include），并将 `.lib` 静态链接库注入系统，无需手动配置复杂的环境变量。

### 5.3 编译与运行指南 (Visual Studio 环境)

1. **克隆/解压源码**：将本项目的所有 `.cpp` 与 `.h` 文件放置在同一工程目录下。
2. **创建项目**：打开 Visual Studio，新建一个 **C++ 空项目 (Empty Project)**。
3. **导入文件**：在“解决方案资源管理器”中，右键点击项目，选择 `添加 -> 现有项`，一口气将所有源码文件全部导入工程。
4. **调整配置**：
   * 在 VS 顶部工具栏将编译模式切换为 **`Release`**（强烈推荐，优化后的时空 A* 搜索速度将提升数倍）。
   * 右键项目进入 `属性 -> 配置属性 -> 一般 -> C++ 语言标准`，将其显式修改为 **`ISO C++17 标准 (/std:c++17)`** 或更高版本。
   * 进入 `属性 -> 配置属性 -> 高级 -> 字符集`，确保配置为 **`使用 Unicode 字符集`**。
5. **编译运行**：按下 `Ctrl + F5` 键执行编译并免调试运行，即可瞬间唤醒 20x20 智能仓储大盘交互视窗。

### 6.1 系统图形界面（GUI）分区布局

   * 压力测试模式完成后的动态刷新核心效能指标：系统总吞吐量（Throughput）与平均订单生存周期（Avg Order Lifecycle），以及每辆 AGV 的当前实时行驶速度。
